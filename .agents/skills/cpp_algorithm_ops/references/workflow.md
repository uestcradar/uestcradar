# 📖 四步防御性部署与测试熔断工作流

本指南详细规定了 C++ 算子在多节点分布式集群环境下的安全发布流程。该流程包含 **“配置读取-环境隔离编译-单测前置熔断-物理分发与热重载”** 四个核心阶段，以 CI 自动化流水线的思想保障线上运行系统零丢包、零故障。

---

## 📌 第一步：定义与解析环境变量

在部署前，编译脚本必须自动从 `cycore` 的本地集群配置文件中解析出目标物理节点的连接与架构参数：

* `REMOTE_IP`：目标物理机 IP（如 `192.162.2.32`）。
* `REMOTE_USER`：SSH 登录用户（默认 `root`）。
* `REMOTE_PASS` | `SSHPASS`：用于 `sshpass` 连接的加密凭证。
* `USE_DOCKER`：若设为 `true`，将在第四步自动重启宿主机容器以热重载。
* `DOCKER_NAME`：目标运行容器名称（如 `sim-du-node2`）。
* `CODE_DIR`：目标机上的 cycore 部署根目录（用于定位 plugins 映射路径）。
* `ARCH`：目标 CPU 硬件架构（如 `arm` 代表 ARM64 主板，`x86` 代表 PC 主机）。

---

## 📌 第二步：基于架构选择 Dockerfile 隔离编译

为了防御 AArch64 编译时由于静态链接外部库导致的 Relocation 链接崩溃，以及 amd64 在 ARM 平台运行的格式报错：

1. **容器环境选择**：
   * 若目标 `ARCH=x86`，编译容器将加载宿主机的 x86 镜像 `cycore-build`；
   * 若目标 `ARCH=arm`，将自适应加载 AArch64 原生或交叉编译镜像；
2. **纯净编译**：
   将 `uestcradar/cpp` 目录挂载进编译容器，在内部运行 CMake 构建，强制生成适合当前物理架构的 `.so`。

---

## 📌 第三步：单测前置熔断 (Test-First Shield)

为了防范带病代码上线，在编译容器中必须首先强制执行算子自带的单元测试。

1. **测试目标执行**：
   在编译容器中，对新编译生成的单元测试二进制程序（如 `qa_fft_block`）发起执行。
2. 🛑 **熔断熔断机制**：
   * **断言失败 (Exit Code != 0)**：
     **必须立即熔断！** 部署程序抛出 `[FATAL] Integration Test Failed!`，完整保留远端物理编译目录与 core dump 现场以供开发者排查，**严禁将动态库拷贝入插件目录**！
   * **断言成功 (Exit Code == 0)**：
     测试通过，输出 `TEST PASSED`，流程被准许进入第四步。

---

## 📌 第四步：物理分发与容器热重载

1. **拷贝覆盖**：
   通过物理通道将 `.so` 复制到目标机 `CODE_DIR/lib/du/flowgraph/plugins/` 挂载路径下。
2. **重启热加载**：
   若 `USE_DOCKER=true`，触发宿主机指令：
   ```bash
   docker restart ${DOCKER_NAME}
   ```
   重启后，流图引擎自动载入最新覆盖的算子，完成热重载。
